<objective>
## Goal
Auditer les fichiers hérités de l'équipe précédente pour identifier ce qui est réutilisable, ce qui est suspect, et ce qui manque.

## Purpose
L'équipe précédente a été licenciée pour suspicion de compromission. Avant tout déploiement, valider l'intégrité du code et des données héritées pour éviter d'embarquer du code malveillant ou corrompu.

## Output
- `reports/heritage_audit.md` : rapport d'audit des fichiers hérités (code, configs, logs, notes)
</objective>

<context>
@.paul/PROJECT.md
@.paul/ROADMAP.md
</context>

<acceptance_criteria>

## AC-1: Structure des fichiers hérités connue
```gherkin
Given les fichiers hérités dans le répertoire du projet
When on liste et inspecte leur contenu
Then on a une carte complète de ce qui existe (code, configs, models, datasets, logs, notes)
```

## AC-2: Éléments suspects identifiés
```gherkin
Given les fichiers de logs et notes personnelles mentionnés dans le sujet
When on les inspecte
Then on a identifié tout contenu suspect, hardcodé ou anormal (secrets, backdoors, données sensibles)
```

## AC-3: Décision sur réutilisabilité
```gherkin
Given l'audit complet
When on analyse le code existant (chatbot, scripts LoRA, configs)
Then on a décidé quels fichiers sont réutilisables as-is, à modifier, ou à ignorer
```

</acceptance_criteria>
