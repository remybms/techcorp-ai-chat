<objective>
## Goal
Tester le chat end-to-end sur des cas financiers réels, corriger les bugs d'intégration, et optimiser les paramètres d'inférence Ollama pour des réponses de qualité.

## Purpose
Un chat qui "marche" techniquement peut produire des réponses lentes, tronquées ou hors-sujet. Ce plan valide la qualité fonctionnelle et optimise les paramètres avant la démo.

## Output
- `scripts/test_api.sh` : suite de tests curl sur l'API Ollama
- `frontend/app.js` : paramètres d'inférence optimisés
- `reports/integration_test_report.md` : résultats de tests
</objective>

## AC-1: Tests E2E passent sur cas financiers
```gherkin
Given le chat opérationnel et Ollama démarré
When on pose 5 questions financières types (P/E ratio, action, dividende, portefeuille, indice)
Then les 5 reçoivent une réponse cohérente, non vide, en moins de 60s chacune
```

## AC-2: Gestion des cas limites
```gherkin
Given l'interface chat
When on envoie un message vide, un message très long (>500 chars), ou une question hors-domaine
Then l'interface gère sans crash : message vide ignoré, long message tronqué ou accepté, hors-domaine répond quand même
```

## AC-3: Paramètres inférence documentés
```gherkin
Given l'API Ollama
When on envoie une requête avec les paramètres optimisés (temperature, num_predict)
Then les réponses sont plus concises et pertinentes qu'avec les valeurs par défaut
```

<task type="auto">
  <name>Task 2: Optimiser les paramètres d'inférence et corriger les cas limites</name>
  <files>frontend/app.js</files>
  <action>
    Dans frontend/app.js, modifier le body de la requête fetch pour ajouter les options Ollama :
    {
      model: MODEL_NAME,
      prompt: prompt,
      stream: true,
      options: {
        temperature: 0.3,      // Déterministe pour finance (vs 0.8 défaut)
        num_predict: 512,       // Limiter la longueur des réponses
        top_p: 0.9,
        repeat_penalty: 1.1
      }
    }

    Corriger cas limites dans sendMessage() :
    - Message vide : if (!prompt || prompt.trim().length === 0) return — déjà géré, vérifier
    - Message très long (>1000 chars) : tronquer à 1000 avec alerte console.warn
    - Timeout : ajouter AbortController avec 60s timeout sur fetch, afficher message si timeout

    Tester manuellement l'interface avec les 5 mêmes questions.
    Comparer qualité/vitesse avec/sans paramètres optimisés dans integration_test_report.md.
  </action>
  <verify>Envoyer "What is a stock?" dans l'interface — réponse arrive en moins de 30s, est concise et financièrement correcte</verify>
  <done>AC-2 et AC-3 satisfaits : cas limites gérés, paramètres optimisés documentés</done>
</task>
